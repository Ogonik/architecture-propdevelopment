# Задание №3.

## Список требований для реализации функциональности Умного дома с применением сервисов партнера

### Общее описание интеграционных процессов

В рамках выполнения бизнес требований по реализации функциональности умного дома реализуется 2 группы фичей:

- интеллектуальный шлагбаум (УШ)
- интеллектуальный домофон (УД)

Согласно описанию требований для реализации будут использоваться сервисы Партнера.
Предположим, что сервисы Партнера реализованы таким образом, что работают почти полностью автономно, наподобие поставщиков услуг ЖКХ. Для обеспечения работы функций шлагбаума и домофона внешним сервисам необходимо иметь информацию о:

- идентификаторе Клиента,
- данных биометрии допущенных лиц Клиента
- данных о ГРН ТС, допущенных на парковку объекта.

#### **Архитектура новой функциональности**

см. диаграмму контейнеров С4.

Архитектура новой функциональности по обеспечению функций умного дома состоит из следующих основополагающих решений:

1. Реализация функциональности ЛК Собственника по управлению услугами УШ и УД в сервисе tenant-core-app: включение/отключение услуг, ввод/обновление информации, необходимой для реализации услуг (**ВНИМАНИЕ! сбор ПДн пользователей класса Собственник и других людей в виде фотографии**)
2. Доработка tenant-core-app для реализации биллинга услуг УШ И УД
3. Создание сервиса-адаптера для обеспечения передачи данных в сервисы Партнера, прием данных по начислениям для дальнейшей оплаты Партнеру. Создается именно новый сервис в целях "развязывания" основного приложения и интеграционных функций, зависящих от протокола взаимодействия с Партнером, который может меняться во времени.
4. Реализация функциональности бухучета для взаиморасчетов с Партнером
5. Реализация функциональности сбора данных в DWH по услугам УД, УШ
6. Разворачивание/Перенос сервисов работы с ПДн в сертифицированное облако (Снижение затрат на инфраструктуру работы с ПДн)
7. Реализация передачи информации сервисам Партнера должна быть выполнена в соответствии с требованиями законодательства по обработке ПДн --> организация vpn-туннеля с ГОСТ шифрованием между сетями PropDevelopment и Партнера.

#### **Точки вызова API Партнера**

- Аутентификация/авторизация
- CRUD сущности Собственник
- СRUD сущности Допущенное лицо (в том числе с фотографиями)
- CRUD сущности Допущенный автомобиль
- CRUD callback экндпоинтов для оповещений о событиях

#### **Точки вызова API Propdevelopment**

- аутентификация/авторизация
- Получение информации о проходе через домофон
- Получение информации о проезде через шлагбаум

### Требования к безопасности

#### **Протоколы аутентификации**

На обоих сторонах (Партнера и PropDevelopment) 
1. Аутентификация с помощью технических учетных записей (ТУЗ), с парой логин-пароль, выдачей JWT токена.
2. На каждой точке вызова проверка JWT-токена в Keycloak, проверка ролей ТУЗ токена
3. Обязательное использование TLS соединений с версией 1.2+
